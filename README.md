### 1. 本部の企業サポート部向け　融資決裁権限に関する問い合わせアシスタントAIの開発
- **期間**：2024年11月~2025年3月（5カ月）
- **体制**：エンジニア2名(QAアプリケーションの設計・開発・運用を担当）
- **対象**：企業サポート部の問い合わせ担当者5名
- **ビジネス課題**：月80件に及ぶ融資決裁権限に関する問い合わせ対応の負荷軽減と回答品質の標準化

#### 背景概要
　銀行業務における融資実行の可否を決定する**決裁権限マニュアルは極めて複雑であり、融資案件が支店長等の裁量内に該当するか、もしくは本部への正式な稟議が必要かを詳述している。融資には細心の注意が必要な上、こちらのマニュアルを読んでも解決しないケースが多いため、本部担当者への問い合わせがチャットボットを通じて頻発していた。また、現場からはマニュアルに記載のない細かな判断基準や回答のニュアンス(Preference)を維持したいという強い要望があった。

#### 役割、アクション
- **データ前処理と匿名化**：過去5,000ペアのQAセットから個人情報（PII）をマスキングし、CSV形式（項目、質問、回答）へ整形。回答列を「回答・条件・参照」に構造化し、LLMが解釈しやすいフォーマットを定義。
- **RAG基盤の構築**：ChromaDBを用いたベクトルデータベースおよびLangChainによるRAGシステムを設計。
- **Human-in-the-Loop(HITL)の実装**：AIの回答をそのまま出力せず、人間が検証・修正を行うプロセスを確立。評価や責任の分離のためだけでなく、「新規の質問に対するAI回答と、それに対する人間の最終回答」をペアで保管・蓄積するデータパイプラインで運用中。
- **継続的学習サイクルの確立**：蓄積された正解データを用いて定期的にベクトルデータベースを更新し、検索精度と回答品質を継続的に向上させるスキームを実装。
- **UI開発**：ipywidgetsを用いたアプリケーションUIの構築。
  
#### 成果
　担当者評価でAccuracy80%を達成。検索時間の短縮に加え、熟練者の言葉遣いを参考にした回答が可能になり、ナレッジの平準化と工数削減に貢献。


#### Technical Stack
RAG, LangChain, ChromaDB, ipywidgets

#### 役割、アクション

### **5. 大手飲料メーカー向け BigQueryを用いたブランダマスタ紐付けPoC(BQML/RAG)**
- **期間**：2025年12月~2026年1月（2カ月）
- **体制**：エンジニア1名、PM1名
- **対象**：大手飲料メーカー
- **ビジネス課題**：既存システムのインフラ複雑性の排除、運用コスト・開発工数の削減（実現可能性の検証）

#### **背景概要**

　顧客先では、Google Cloud上でCloud FunctionやWorkflowsなど用いたマスタの紐付けプロジェクトのインフラの複雑性や開発・運用コストを懸念している。SQLベースでAI機能を利用できるBQMLや定期スケジュール実行などのBig Query内部機能を用いることで、従前のシステムの複雑性を排除したRAGベースの**商品名とブランド名の紐付けタスクの実現可能性を検証、提案するため本PoCに着手。

#### **役割、アクション**
- Google Cloud StorageからBig Queryへのデータのロード
- データ前処理：商品名の略称に対応するための拡張データを作成。NFCK正規化や文字列置換、ユニーク化を行うUDF（ユーザー定義関数）を実装し、AI処理前に商品名のユニーク数を80%削減
- AI処理前の完全一致結合
- Remote model機能を用いて、埋め込みモデル(gemini-embeddig-001)で商品名とブランド名をベクトル化したテーブルの作成
- ベクトル検索で算出された、コサイン距離の近い上位５件をプロンプトに組み込ませ、テキスト生成モデル(gemini-flash-2.5)でブランド名を判断理由とともにJSON形式で推定(LLM機能はデフォルトでバッチ処理）
- プロシージャとして上記の一連のSQLをワークフローとして定義し、定期実行スケジュールの作成
  
#### **成果**
1,500件の商品データに対して正解率80%、実行時間1分半、コスト数十円とったパフォーマンスを実証。また、Big QueryのみでRAGを実現することでインフラ管理不要化による運用工数の削減を証明。

#### **Technical Stack**
SQL, BQML, Google Cloud, Big Query
