## 職務概要

- **専門性・クラウド**：Googleの[Advanced Solutions Lab](https://cloud.google.com/customers/chiba-bank?hl=ja)を修了。複数のAWS、Google Cloudの認定資格を取得。学生時代から取り組んでいるKaggleやSIGNATEで培った実装力やドメインをビジネス課題に応用。

### 1. 本部の企業サポート部向け　融資決裁権限に関する問い合わせアシスタントAIの開発
- **期間**：2024年11月~2025年3月（5カ月）
- **体制**：エンジニア2名
- **対象**：企業サポート部の問い合わせ担当者5名
- **ビジネス課題**：月80件に及ぶ融資決裁権限に関する問い合わせ対応の負荷軽減と回答品質の標準化

#### 背景概要
　銀行では、融資実行の可否を決定する**決裁権限**に関するマニュアルがあり、融資案件が支店長等の裁量内に該当するか、もしくは本部への正式な稟議が必要かを詳述している。融資には細心の注意が必要な上、こちらのマニュアルを読んでも解決しないケースが多いため、本部担当者への問い合わせがチャットボットを通じて頻発していた。開発にあたって、業務側からは、マニュアルに記載のない細かな判断基準や回答のニュアンスを維持したいという要望があった。

#### 役割、アクション
- **データ前処理と匿名化**：過去5,000ペアのQAセットから個人情報をマスキングし、CSV形式（項目、質問、回答）へ整形。回答列を「回答・条件・参照」に構造化し、LLMが解釈しやすいフォーマットを定義。
- **RAG基盤の構築**：ChromaDBを用いたベクトルデータベースおよびLangChainによるRAGシステムを設計。
- **Human-in-the-Loop(HITL)の実装**：AIの回答をそのまま出力せず、人間が検証・修正を行うプロセスを確立。評価や責任の分離のためだけでなく、新規の質問に対するAIと人間の最終回答をペアで保管・蓄積するデータパイプラインで運用中。
- **学習サイクルの担当**：回答品質を継続的に向上させるため、蓄積された正解データを用いて定期的にベクトルデータベースを更新。
- **UI開発**：ipywidgetsを用いたアプリケーションUIの構築。
  
#### 成果
　担当者評価でAccuracy80%を達成。検索時間の短縮に加え、熟練者の言葉遣いを参考にした回答が可能になり、ナレッジの平準化と工数削減に貢献。


#### Technical Stack
RAG, LangChain, ChromaDB, ipywidgets

### **2. 市場業務部向け　国際収支項目番号検索AI(外国送金マニュアルのRAGシステム)**

- **期間**：2024年11月～2025年3月（5カ月）
- **体制**：エンジニア2名
- **対象**：市場業務部の行員（8名程度）
- **ビジネス課題**：80ページに及ぶ日本銀行発行マニュアルの参照工数の削減、および本部への業務集中に伴う負荷軽減

#### **背景概要**
　市場業務部では、送金目的に応じて指定される、3桁の国際収支項目番号を日本銀行へ報告する必要がある外国送金業務を取り扱っており、発行された80ページものマニュアルを参照しており、多くの時間を要していた。中期経営計画で掲げる意思決定や情報の本部集中に伴う業務増に対応するため、新任スタッフでも、迅速に過去ケースやマニュアルを参照できる環境が必要であった。

#### **役割、アクション**
- **E2E**：業務課題のヒアリングから要件定義、開発、デプロイまで一貫して担当。
- **PDFパースの最適化**：マニュアルの階層構造に基づいたチャンク化を実施し、検索精度を向上。
- **トレーサビリティの確保**：回答の根拠となったPDFの該当箇所をStreamlit上で即座にプレビューできるメタデータ連携機能を実装。
- **コンテナ運用**：Google CloudのCloud Runを用いたサーバーレスなデプロイ・運用環境を構築。

#### **成果**
　担当者評価でAccuracy80%（Correct/Helpful/Incorrect指標）を達成。また、マニュアル参照時間を大幅に削減。

#### **Technical Stack**
Cloud Run, GCS, Streamlit, RAG, LangChain, ChromaDB

### **3. デジタル戦略部業務改革G向け　提案制度振り分けAIエージェント(RAG/AI-agent)**
- **期間**：2025年3月～2025年5月（3カ月）
- **体制**：エンジニア1名 
- **対象**：提案制度システム受付担当者(毎月2名)
- **ビジネス課題**：過去4,000件に及ぶ改善提案に対する重複判定と担当部署割り当て作業の効率化

#### **背景概要**
　デジタル戦略部業務改革グループでは、支店行員から日常業務の改善案を受け付けている掲示板を４年ほど運用しており、過去4,000件の提案がある。4年間の運用で蓄積された膨大なデータに対し、新しい提案の重複チェックや適切な対応先の判断が属人化し、処理工数がボトルネックとなっていた。
　
#### 役割、アクション
- **要件定義とプロセス設計**：業務改革グループへの直接ヒアリングを通じ、従前の判断フローをAIエージェントで代替。
- **マルチエージェントの実装**：LangGraphを活用し、重複判定と所管部振り分けを個別のエージェントに分担させるワークフローを構築。
- **データセットの整形と匿名化**：CSVカラムの整形と過去4,000件のデータ内の個人情報のマスキング
- **差分更新インデックスの実装**：担当者がUI上で新規提案を追加する際、ベクトルDBのインデックス全体を再構築するのではなく、追加された差分のみを処理する仕組みを実装し、データベース更新時の工数を最小化。
- **フルスタック開発**：Streamlitを用いたUI構築、Cloud Runへのデプロイ、GCSおよび一時データとの連携処理を完遂。

#### **成果**
　1件あたりの処理時間を平均15分から約1分へ短縮（93%削減）。年間約1,000件の提案に対し、約233時間の工数削減を達成。また、類似検索の観点で新規スタッフ層からの定性的なポジティブな評価を獲得。

#### **Technical Stack**
LangGraph LangChain, RAG, ChromaDB, GCS, Cloud Run, Artifact Registry, Streamlit

### **5. 大手飲料メーカー向け BigQueryを用いたブランダマスタ紐付けPoC(BQML/RAG)**
- **期間**：2025年12月~2026年1月（2カ月）
- **体制**：エンジニア1名、PM1名
- **対象**：大手飲料メーカー
- **ビジネス課題**：既存システムのインフラ複雑性の排除、運用コスト・開発工数の削減（実現可能性の検証）

#### **背景概要**

　顧客先では、Google Cloud上でCloud FunctionやWorkflowsなど用いたマスタの紐付けシステムにおけるインフラの複雑性や開発・運用コストを懸念している。SQLベースでAI機能を利用できるBQMLや定期スケジュール実行などのBig Query内部機能を用いることで、従前のシステムの複雑性を排除したRAGベースの商品名とブランド名の紐付けタスクの実現可能性を検証、提案するため本PoCに着手。

#### **役割、アクション**
- Google Cloud StorageからBig Queryへのデータのロード
- データ前処理：商品名の略称に対応するための拡張データを作成。NFCK正規化や文字列置換、ユニーク化を行うUDF（ユーザー定義関数）を実装し、AI処理前に商品名のユニーク数を80%削減
- AI処理前の完全一致結合
- Remote model機能を用いて、埋め込みモデルであるgemini-embeddig-001で商品名とブランド名をベクトル化したテーブルの作成
- ベクトル検索で算出された、コサイン距離の近い上位５件をプロンプトに組み込ませ、テキスト生成モデルであるgemini-flash-2.5でブランド名を判断理由とともにJSON形式で推定(デフォルトでバッチ処理）
- プロシージャとして上記の一連のSQLをワークフローとして定義し、定期実行スケジュールの実証。
  
#### **成果**
1,500件の商品データに対して正解率80%、実行時間1分半、コスト数十円とったパフォーマンスを実証。また、Big QueryのみでRAGを実現することでインフラ管理不要化による運用工数の削減を証明。

#### **Technical Stack**
SQL, BQML, Google Cloud, Big Query
